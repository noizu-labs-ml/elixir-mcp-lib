## 1. Stateless Protocol Core

The protocol is now **fully stateless**. This is the single largest architectural change.

### What was removed

| Removed                                    | Replacement                                            |
|--------------------------------------------|--------------------------------------------------------|
| `initialize` / `notifications/initialized` | Per-request `_meta` fields on every request            |
| `Mcp-Session-Id` header                    | Eliminated entirely; no protocol-level sessions        |
| HTTP GET endpoint (SSE stream)             | `subscriptions/listen` POST request                    |
| `ping` method                              | Eliminated                                             |
| `logging/setLevel` method                  | `io.modelcontextprotocol/logLevel` in `_meta`          |
| `notifications/roots/list_changed`         | MRTR pattern for roots                                 |
| `resources/subscribe` / `unsubscribe`      | `subscriptions/listen` with notification filter        |
| Server-initiated JSON-RPC requests on SSE  | MRTR pattern (`InputRequiredResult`)                   |
| `Last-Event-ID` resumable streams          | Eliminated; streams not resumable                      |

### How it works now

Every request is self-contained. The server processes each request independently with
no inferred state from prior requests, even on the same connection.

Key rules:

- Servers MUST NOT rely on prior requests for context (capabilities, version, identity).
- Every request carries version, client info, and capabilities in `_meta`.
- An open connection (stdio process, HTTP connection) is NOT a session.
- Clients MAY interleave unrelated requests on the same transport.
- State spanning multiple requests MUST use explicit identifiers (handles) passed on each request.
- Long-lived requests like `subscriptions/listen` are still request/response; the response is an open stream of notifications scoped to that request.

### Terminology (eras)

| Term       | Meaning                                                        |
|------------|----------------------------------------------------------------|
| **Modern** | `2026-07-28` and later -- per-request metadata, no handshake   |
| **Legacy** | `2025-11-25` and earlier -- `initialize` handshake, sessions   |
| **Dual-era** | Implementation supporting both modern and legacy            |

## 2. `server/discover` Method

Replaces capability negotiation from the old `initialize` response. Servers MUST implement it.

### Request

No body parameters beyond standard `_meta`:

```json
{
  "jsonrpc": "2.0",
  "id": "discover-1",
  "method": "server/discover",
  "params": {
    "_meta": {
      "io.modelcontextprotocol/protocolVersion": "2026-07-28",
      "io.modelcontextprotocol/clientInfo": {
        "name": "ExampleClient",
        "version": "1.0.0"
      },
      "io.modelcontextprotocol/clientCapabilities": {}
    }
  }
}
```

### Response

```json
{
  "jsonrpc": "2.0",
  "id": "discover-1",
  "result": {
    "resultType": "complete",
    "supportedVersions": ["2026-07-28"],
    "capabilities": {
      "tools": {},
      "resources": {}
    },
    "serverInfo": {
      "name": "ExampleServer",
      "version": "1.0.0"
    },
    "instructions": "This server provides weather and resource utilities.",
    "ttlMs": 3600000,
    "cacheScope": "public"
  }
}
```

### Response Fields

| Field               | Type                 | Required | Description                                                 |
|---------------------|----------------------|----------|-------------------------------------------------------------|
| `supportedVersions` | `string[]`           | Yes      | Protocol versions the server supports                       |
| `capabilities`      | `ServerCapabilities` | Yes      | Server capabilities (tools, resources, prompts, extensions) |
| `serverInfo`        | `Implementation`     | Yes      | Name and version of the server                              |
| `instructions`      | `string`             | No       | Natural-language guidance for LLMs on how to use the server |

### When to call

Calling is OPTIONAL. A client MAY invoke any RPC inline and handle
`UnsupportedProtocolVersionError` if the version is wrong. Useful for:

- **Up-front version selection** -- learn supported versions before sending other requests.
- **stdio backward-compatibility probe** -- detect modern vs. legacy server (send `server/discover` first; on non-modern error, fall back to `initialize`).

## 3. Protocol Metadata in `_meta`

Every client request MUST include the following fields in `params._meta`:

| Key                                          | Type                 | Required | Description                                               |
|----------------------------------------------|----------------------|----------|-----------------------------------------------------------|
| `io.modelcontextprotocol/protocolVersion`    | `string`             | Yes      | Protocol version (e.g., `"2026-07-28"`)                   |
| `io.modelcontextprotocol/clientInfo`         | `Implementation`     | Yes      | Client name and version                                   |
| `io.modelcontextprotocol/clientCapabilities` | `ClientCapabilities` | Yes      | Client capabilities for this request                      |
| `io.modelcontextprotocol/logLevel`           | `LoggingLevel`       | No       | Min log level the server should emit for this request     |

A request missing any required field is malformed; the server MUST reject with
JSON-RPC error `-32602` (Invalid params). On HTTP, status MUST be `400 Bad Request`.

If a server needs a capability the client did not declare, it MUST return
`MissingRequiredClientCapabilityError` (`-32003`) with `data.requiredCapabilities`
listing the missing capabilities.

### `_meta` key naming rules

- Format: optional **prefix** + **name**.
- Prefix: reverse-DNS followed by `/` (e.g., `io.modelcontextprotocol/`, `com.example/`).
- Any prefix where the second label is `modelcontextprotocol` or `mcp` is reserved for MCP use.
- Third-party extensions use their own reverse-DNS prefix.

### Example: full `_meta` on a `tools/call` request

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": { "location": "New York" },
    "_meta": {
      "io.modelcontextprotocol/protocolVersion": "2026-07-28",
      "io.modelcontextprotocol/clientInfo": {
        "name": "ExampleClient",
        "version": "1.0.0"
      },
      "io.modelcontextprotocol/clientCapabilities": {
        "elicitation": {},
        "extensions": {
          "io.modelcontextprotocol/tasks": {}
        }
      },
      "io.modelcontextprotocol/logLevel": "warning",
      "traceparent": "00-0af7651916cd43dd8448eb211c80319c-00f067aa0ba902b7-01"
    }
  }
}
```

## 4. Server-to-Client Request Restrictions

**Breaking change.** Servers can no longer send independent JSON-RPC requests to clients
on SSE streams. The old pattern (server sends `sampling/createMessage`, `elicitation/create`,
`roots/list` as standalone requests on the SSE stream) is eliminated.

All server-to-client interactions are now embedded inside results as `InputRequiredResult`
via the MRTR pattern (see next section). This means:

- The server MUST NOT send independent JSON-RPC requests on any stream.
- Request-scoped notifications (`notifications/progress`, `notifications/message`) still
  flow on the SSE response stream of the request they relate to.
- Long-lived notifications (`tools/list_changed`, etc.) flow on `subscriptions/listen`.

## 5. Multi Round-Trip Requests (MRTR)

Replaces server-initiated requests entirely. SEP-2322.

### Core flow

1. Client sends a request (e.g., `tools/call`).
2. Server needs more info -- returns `InputRequiredResult` with `resultType: "input_required"`.
3. Client gathers the requested info, retries the original request with `inputResponses` + `requestState`.
4. Server completes the request and returns the final result.

### `InputRequiredResult`

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "resultType": "input_required",
    "inputRequests": {
      "github_login": {
        "method": "elicitation/create",
        "params": {
          "mode": "form",
          "message": "Please provide your GitHub username",
          "requestedSchema": {
            "type": "object",
            "properties": {
              "name": { "type": "string" }
            },
            "required": ["name"]
          }
        }
      },
      "capital_of_france": {
        "method": "sampling/createMessage",
        "params": {
          "messages": [
            {
              "role": "user",
              "content": { "type": "text", "text": "What is the capital of France?" }
            }
          ],
          "systemPrompt": "You are a helpful assistant.",
          "maxTokens": 100
        }
      }
    },
    "requestState": "AEAD-protected blob"
  }
}
```

### `InputResponses` (client retry)

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": { "location": "New York" },
    "inputResponses": {
      "github_login": {
        "action": "accept",
        "content": { "name": "octocat" }
      },
      "capital_of_france": {
        "role": "assistant",
        "content": { "type": "text", "text": "The capital of France is Paris." },
        "model": "claude-3-sonnet-20240307",
        "stopReason": "endTurn"
      }
    },
    "requestState": "AEAD-protected blob",
    "_meta": {
      "io.modelcontextprotocol/protocolVersion": "2026-07-28",
      "io.modelcontextprotocol/clientInfo": { "name": "ExampleClient", "version": "1.0.0" },
      "io.modelcontextprotocol/clientCapabilities": { "elicitation": {} }
    }
  }
}
```

### `resultType` values

| Value              | Meaning                                                                 |
|--------------------|-------------------------------------------------------------------------|
| `"complete"`       | Request completed successfully, result contains final content           |
| `"input_required"` | More info needed; result contains `InputRequiredResult`                 |
| `"task"`           | Tasks extension: long-running operation, result is `CreateTaskResult`   |
| Absent             | Backward compat -- treat as `"complete"` for legacy servers             |
| Unrecognized       | MUST be considered invalid                                              |

### Supported requests (that may return `InputRequiredResult`)

| Client Request     | Supports InputRequiredResult |
|--------------------|------------------------------|
| `prompts/get`      | Yes                          |
| `resources/read`   | Yes                          |
| `tools/call`       | Yes                          |

All other requests MUST NOT return `InputRequiredResult`.

### `requestState` security

- Servers MUST treat `requestState` as attacker-controlled input.
- If it influences authorization or business logic, servers MUST protect integrity (HMAC or AEAD).
- Servers SHOULD include: authenticated principal, short TTL, originating request identifier.
- Clients MUST NOT inspect, parse, or modify `requestState`.

## 6. Routing Headers

New required HTTP headers on Streamable HTTP POST requests for intermediary routing.

### Protocol Version Header

Every POST MUST include: `MCP-Protocol-Version: 2026-07-28`

Must match `io.modelcontextprotocol/protocolVersion` in `_meta`. Mismatch = `400 Bad Request`.

### Standard Request Headers

| Header       | Source Field                  | Required For                                   |
|--------------|-------------------------------|------------------------------------------------|
| `Mcp-Method` | `method`                      | All requests and notifications                 |
| `Mcp-Name`   | `params.name` or `params.uri` | `tools/call`, `resources/read`, `prompts/get`  |

### Example: `tools/call`

```http
POST /mcp HTTP/1.1
Content-Type: application/json
MCP-Protocol-Version: 2026-07-28
Mcp-Method: tools/call
Mcp-Name: get_weather

{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": { "location": "Seattle, WA" },
    "_meta": {
      "io.modelcontextprotocol/protocolVersion": "2026-07-28",
      "io.modelcontextprotocol/clientInfo": { "name": "ExampleClient", "version": "1.0.0" },
      "io.modelcontextprotocol/clientCapabilities": {}
    }
  }
}
```

### Custom Headers from Tool Parameters (`x-mcp-header`)

Servers MAY annotate tool input properties with `x-mcp-header` to mirror values into
HTTP headers as `Mcp-Param-{Name}`.

```json
{
  "name": "execute_sql",
  "description": "Execute SQL on Google Cloud Spanner",
  "inputSchema": {
    "type": "object",
    "properties": {
      "region": {
        "type": "string",
        "description": "The region to execute the query in",
        "x-mcp-header": "Region"
      },
      "query": {
        "type": "string",
        "description": "The SQL query to execute"
      }
    },
    "required": ["region", "query"]
  }
}
```

Results in header: `Mcp-Param-Region: us-west1`

Constraints on `x-mcp-header`:
- MUST NOT be empty
- MUST match HTTP field-name token syntax
- MUST be case-insensitively unique within the `inputSchema`
- Only primitive types (string, integer, boolean) -- NOT `number`
- Non-ASCII values use Base64 encoding: `=?base64?{encoded}?=`

Clients on Streamable HTTP MUST reject tools with invalid `x-mcp-header`. Clients on
stdio MAY ignore them.

### Server Validation

Servers MUST validate header values match body values. Mismatch = `400 Bad Request` with
JSON-RPC error code `-32001` (`HeaderMismatch`).

## 7. Caching Metadata

New `CacheableResult` fields MUST be present on results from these operations:

- `server/discover`
- `tools/list`
- `prompts/list`
- `resources/list`
- `resources/templates/list`
- `resources/read`

### Fields

| Field        | Type                      | Description                                          |
|--------------|---------------------------|------------------------------------------------------|
| `ttlMs`      | `integer` (milliseconds)  | Freshness hint; analogous to HTTP `max-age`          |
| `cacheScope` | `"public"` or `"private"` | Whether shared intermediaries may cache the response  |

### `ttlMs` semantics

- `0` = immediately stale (re-fetch every time).
- Positive = fresh for that many ms after `t_received`.
- Absent = assume `0` (legacy servers only).
- Negative = treat as `0`.
- Servers MUST provide `ttlMs >= 0`.

### `cacheScope` semantics

| Value       | Meaning                                                                    |
|-------------|----------------------------------------------------------------------------|
| `"public"`  | No user-specific data. Any cache or proxy MAY store and serve to any user. |
| `"private"` | Private data. Reuse only within same authorization context.                |

### Example

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "resultType": "complete",
    "tools": [
      {
        "name": "get_weather",
        "description": "Get current weather",
        "inputSchema": { "type": "object", "properties": { "location": { "type": "string" } } }
      }
    ],
    "ttlMs": 300000,
    "cacheScope": "public"
  }
}
```

### Interaction with notifications

- TTL and `listChanged` notifications are complementary.
- A notification received while cache is fresh immediately invalidates the cache.
- Clients SHOULD NOT use TTL as a polling interval; check freshness on access.

## 8. W3C Trace Context in `_meta`

Reserved keys in `_meta` for OpenTelemetry trace context propagation (SEP-414):

| Key           | Format                | Spec                     |
|---------------|-----------------------|--------------------------|
| `traceparent` | W3C Trace Context     | https://www.w3.org/TR/trace-context/ |

## 9. MCP Apps

Extension ID: `io.modelcontextprotocol/ui`

Server-rendered interactive HTML UIs that render inline in conversations within
sandboxed iframes. Uses its own dialect of MCP over `postMessage` transport.

### How it works

1. **UI preloading**: Tool description includes `_meta.ui.resourceUri` pointing to a `ui://` resource.
2. **Resource fetch**: Host fetches the HTML UI resource from the server.
3. **Sandboxed rendering**: HTML rendered in a sandboxed iframe. No access to parent DOM,
   cookies, localStorage, or navigation.
4. **Bidirectional communication**: JSON-RPC over `postMessage`. App can call MCP tools,
   send messages, update model context. Host can push data to the app.

### Client capability declaration

```json
{
  "io.modelcontextprotocol/clientCapabilities": {
    "extensions": {
      "io.modelcontextprotocol/ui": {
        "mimeTypes": ["text/html;profile=mcp-app"]
      }
    }
  }
}
```

### Security model

- Runs in sandboxed iframe
- Cannot access parent window DOM, cookies, storage
- All communication via `postMessage`
- Host controls which capabilities the app can access
- Apps can include `permissions` (microphone, camera) and `csp` directives

### Supported clients (as of RC)

Claude, Claude Desktop, VS Code GitHub Copilot, Goose, Postman, MCPJam, Archestra.AI.

## 10. Tasks Extension

Extension ID: `io.modelcontextprotocol/tasks`

Promoted from experimental to official extension (SEP-2663). Redesigned with handle-based
lifecycle.

### Changes from experimental

| Old (experimental)           | New (official extension)                               |
|------------------------------|--------------------------------------------------------|
| `tasks/result` (blocking)    | `tasks/get` (polling)                                  |
| `tasks/list`                 | Removed                                                |
| Per-request opt-in           | Server can return task handles unsolicited (if client declared capability) |
| Part of core protocol        | Extension: `io.modelcontextprotocol/tasks`             |

### Task lifecycle states

| Status           | Meaning                                                    |
|------------------|------------------------------------------------------------|
| `working`        | Operation in progress                                      |
| `input_required` | Server needs client input before continuing                |
| `completed`      | Done; `result` field has the output                        |
| `failed`         | Error; `error` field has JSON-RPC error details            |
| `cancelled`      | Cancelled (cooperative, may still reach another terminal)  |

`completed`, `failed`, `cancelled` are terminal.

### Capability negotiation

Client declares in per-request `_meta`:

```json
{
  "io.modelcontextprotocol/clientCapabilities": {
    "extensions": {
      "io.modelcontextprotocol/tasks": {}
    }
  }
}
```

Server advertises in `server/discover` response:

```json
{
  "capabilities": {
    "tools": {},
    "extensions": {
      "io.modelcontextprotocol/tasks": {}
    }
  }
}
```

### `CreateTaskResult` (`resultType: "task"`)

Returned instead of the final result when the server decides a request will be long-running:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "resultType": "task",
    "taskId": "task-abc-123",
    "status": "working",
    "ttlMs": 3600000,
    "pollIntervalMs": 5000
  }
}
```

### Methods

**`tasks/get`** -- poll for task status:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "method": "tasks/get",
  "params": {
    "taskId": "task-abc-123",
    "_meta": { "..." : "..." }
  }
}
```

Response when completed:

```json
{
  "jsonrpc": "2.0",
  "id": 2,
  "result": {
    "resultType": "complete",
    "taskId": "task-abc-123",
    "status": "completed",
    "result": {
      "content": [{ "type": "text", "text": "Build succeeded" }]
    }
  }
}
```

**`tasks/update`** -- provide input when `input_required`:

```json
{
  "jsonrpc": "2.0",
  "id": 3,
  "method": "tasks/update",
  "params": {
    "taskId": "task-abc-123",
    "inputResponses": {
      "confirm_deploy": { "action": "accept", "content": { "approved": true } }
    },
    "_meta": { "..." : "..." }
  }
}
```

**`tasks/cancel`** -- cooperative cancellation:

```json
{
  "jsonrpc": "2.0",
  "id": 4,
  "method": "tasks/cancel",
  "params": {
    "taskId": "task-abc-123",
    "_meta": { "..." : "..." }
  }
}
```

### Notifications

Servers can push `notifications/tasks` via `subscriptions/listen`. Each notification
carries full task state, eliminating extra `tasks/get` round-trips.

## 11. Extensions Framework

Formalized as a first-class part of the protocol.

### Extension identifiers

Format: `{vendor-prefix}/{extension-name}`

- MUST follow `_meta` key naming rules with mandatory prefix.
- Official extensions: `io.modelcontextprotocol/` prefix.
- Third-party: use reverse-DNS you own (e.g., `com.example/my-extension`).

### Capability map

Extensions are advertised in the `extensions` field of capabilities:

```json
{
  "capabilities": {
    "tools": {},
    "extensions": {
      "io.modelcontextprotocol/tasks": {},
      "io.modelcontextprotocol/ui": {
        "mimeTypes": ["text/html;profile=mcp-app"]
      },
      "com.mycompany/custom-auth": { "version": "2.0" }
    }
  }
}
```

### Negotiation

- Both client and server declare support in `extensions` within their capabilities.
- If one party supports an extension but the other does not, the supporting party MUST
  either revert to core protocol behavior or reject with an appropriate error.
- Extensions SHOULD document expected fallback behavior.
- Extensions are always disabled by default and require explicit opt-in.

### Extension lifecycle

- SEP-based proposal process (Extensions Track).
- At least one reference SDK implementation required before review.
- Official repos in GitHub org with `ext-` prefix.
- Experimental repos use `experimental-ext-` prefix.
- Extensions evolve independently of core protocol.
- Breaking changes should use new identifier (e.g., `v2` suffix).

## 12. JSON Schema 2020-12 Full Support

SEP-2106. `inputSchema` and `outputSchema` now support the full JSON Schema 2020-12 vocabulary.

### Key changes

- **Default dialect**: When no `$schema` field, defaults to JSON Schema 2020-12.
- **Explicit dialect**: Schemas MAY include `$schema` for a different dialect (e.g., draft-07).
- **Required support**: Implementations MUST support 2020-12 for schemas without explicit `$schema`.
- **Composition keywords**: `anyOf`, `oneOf`, `allOf`, `if`/`then`/`else`, `$defs` fully supported.
- **`$ref` resolution**: MUST NOT automatically dereference `$ref` to network URIs. MAY offer opt-in
  with allowlists, timeouts, size limits.
- **`structuredContent`**: Now allows any JSON value (not just objects).

### Resource bounds

Implementations SHOULD apply bounds on composition keywords:
- Maximum schema depth
- Cap on total number of subschemas
- Per-validation time budget

This prevents malicious schemas from acting as DoS vectors.

### Example: tool with explicit draft-07

```json
{
  "name": "calculate_sum",
  "inputSchema": {
    "$schema": "http://json-schema.org/draft-07/schema#",
    "type": "object",
    "properties": {
      "a": { "type": "number" },
      "b": { "type": "number" }
    },
    "required": ["a", "b"]
  }
}
```

### Example: tool with composition keywords (2020-12 default)

```json
{
  "name": "create_entity",
  "inputSchema": {
    "type": "object",
    "$defs": {
      "address": {
        "type": "object",
        "properties": {
          "street": { "type": "string" },
          "city": { "type": "string" }
        },
        "required": ["street", "city"]
      }
    },
    "properties": {
      "type": {
        "type": "string",
        "enum": ["person", "company"]
      },
      "address": { "$ref": "#/$defs/address" }
    },
    "oneOf": [
      {
        "properties": { "type": { "const": "person" }, "age": { "type": "integer" } },
        "required": ["type", "age"]
      },
      {
        "properties": { "type": { "const": "company" }, "employees": { "type": "integer" } },
        "required": ["type", "employees"]
      }
    ]
  }
}
```

## 13. Deprecations

All deprecated features remain functional during a **minimum 12-month** deprecation
window from the `2026-07-28` revision release. New implementations SHOULD NOT adopt them.

### Deprecated features (SEP-2577)

| Feature    | Migration Path                                                               |
|------------|-----------------------------------------------------------------------------|
| **Roots**  | Pass directories/files via tool parameters, resource URIs, or server config |
| **Sampling** | Integrate directly with LLM provider APIs                                |
| **Logging** | Use `stderr` (stdio) or OpenTelemetry                                     |

### Other deprecations

| Feature                               | Since         | Migration                                |
|---------------------------------------|---------------|------------------------------------------|

## 14. Authorization Hardening

### RFC 9207: `iss` parameter validation

Authorization servers SHOULD include `iss` in authorization responses (per RFC 9207).

Client validation rules:

| `authorization_response_iss_parameter_supported` | `iss` present | Action                                     |
|--------------------------------------------------|---------------|--------------------------------------------|
| `true`                                           | Yes           | Compare to recorded issuer (string match)  |
| `true`                                           | No            | Reject the response                        |
| `false` or absent                                | Yes           | Compare to recorded issuer (string match)  |
| `false` or absent                                | No            | Proceed                                    |

Clients MUST NOT apply URI normalization before comparison. Validation applies to error
responses too -- on mismatch, MUST NOT act on or display `error` / `error_description`.

A future revision is expected to upgrade `iss` inclusion from SHOULD to MUST.

### `application_type` during Dynamic Client Registration

Clients MUST specify an appropriate `application_type` to avoid OpenID Connect redirect
URI conflicts (SEP-837).

### Credential binding to authorization server

Clients MUST:
- Key persisted credentials by issuer identifier.
- MUST NOT reuse credentials with a different authorization server.
- MUST re-register when the authorization server changes.

(SEP-2352)

### Client ID Metadata Documents

New preferred registration mechanism (replacing Dynamic Client Registration):
- Client uses an HTTPS URL as `client_id`.
- Authorization server detects URL-formatted `client_id`, fetches metadata from that URL.
- Server validates metadata and `redirect_uris`.

Dynamic Client Registration (RFC 7591) is deprecated but retained for backward
compatibility.

### Other OAuth requirements

- Servers MUST implement OAuth 2.0 Protected Resource Metadata (RFC 9728).
- Clients MUST use Protected Resource Metadata for authorization server discovery.
- Authorization servers MUST provide OAuth 2.0 Authorization Server Metadata (RFC 8414)
  or OpenID Connect Discovery 1.0 (or both).
- Clients MUST support both discovery mechanisms.
- Clients MUST implement Resource Indicators (RFC 8707) with the `resource` parameter
  on authorization and token requests.
- Tokens MUST use `Authorization: Bearer` header, MUST NOT be in URI query string.

## 16. `subscriptions/listen`

Replaces the HTTP GET endpoint, `resources/subscribe`/`unsubscribe`, and
`notifications/roots/list_changed`. SEP-2575.

### Opening a stream

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "subscriptions/listen",
  "params": {
    "_meta": {
      "io.modelcontextprotocol/protocolVersion": "2026-07-28",
      "io.modelcontextprotocol/clientInfo": { "name": "ExampleClient", "version": "1.0.0" },
      "io.modelcontextprotocol/clientCapabilities": {}
    },
    "notifications": {
      "toolsListChanged": true,
      "promptsListChanged": true,
      "resourcesListChanged": true,
      "resourceSubscriptions": ["file:///project/config.json"]
    }
  }
}
```

### Notification filter fields

| Field                   | Type       | Description                                                        |
|-------------------------|------------|--------------------------------------------------------------------|
| `toolsListChanged`      | `boolean`  | Receive `notifications/tools/list_changed`                         |
| `promptsListChanged`    | `boolean`  | Receive `notifications/prompts/list_changed`                       |
| `resourcesListChanged`  | `boolean`  | Receive `notifications/resources/list_changed`                     |
| `resourceSubscriptions` | `string[]` | Receive `notifications/resources/updated` for specified URIs       |

### Server acknowledgment

MUST be the first message on the stream. Reflects the subset the server agreed to honor:

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/subscriptions/acknowledged",
  "params": {
    "_meta": {
      "io.modelcontextprotocol/subscriptionId": "1"
    },
    "notifications": {
      "toolsListChanged": true,
      "resourceSubscriptions": ["file:///project/config.json"]
    }
  }
}
```

### Notification delivery

All notifications carry `io.modelcontextprotocol/subscriptionId` in `_meta`:

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/resources/updated",
  "params": {
    "_meta": {
      "io.modelcontextprotocol/subscriptionId": "1"
    },
    "uri": "file:///project/config.json"
  }
}
```

### Key rules

- Server MUST NOT send notification types the client did not request.
- Request-scoped notifications (`progress`, `message`) flow on the response stream
  of the originating request, NOT on `subscriptions/listen`.
- Multiple concurrent subscriptions are supported; demultiplex via `subscriptionId`.
- On stdio, `notifications/cancelled` with the listen request ID terminates the subscription.
- No state persists across reconnections; client MUST re-send `subscriptions/listen` after reconnect.
